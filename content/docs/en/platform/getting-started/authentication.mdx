Nexus Signal uses **environment-scoped API keys**. Different credentials apply to server triggers vs. browser-facing SDK components.

## Key types

| Key | Prefix | Used by | Exposure |
|-----|--------|---------|----------|
| **Secret key** | `sk_*` | Server SDK, workflow triggers, schedule SDK | **Server only — never in browser** |
| **Public key** | `pk_*` | React SDK, inbox API, WebSocket join | Safe to embed in frontend |

Secret keys are sent as `Authorization: Bearer sk_prd_…` on trigger requests.

Public keys are sent as `x-nexus-public-key` on SDK routes.

## Workflow trigger auth

```http
POST /v1/workflows/trigger
Authorization: Bearer sk_prd_your_secret_key
Content-Type: application/json
```

The API validates the key, resolves the environment and organization, then returns **202 Accepted** while the worker processes delivery asynchronously.

## HMAC for real-time (in-app)

Browser WebSocket connections cannot use secret keys. Instead:

1. **Your backend** generates an HMAC signature with the secret key and subscriber ID
2. **Your frontend** passes the signature to `NexusProvider`

```ts
// Server — @nexus-signal/node
const hmac = nexus.auth.generateHmacSignature('user_123');

// Client — @nexus-signal/react
<NexusProvider
  publicKey="pk_prd_…"
  subscriberId="user_123"
  hmacSignature={hmac}
  backendUrl="https://api.nexussignal.dev"
/>
```

The platform verifies: `HMAC-SHA256(secretKey, subscriberExternalId)`.

<Callout type="error">
Regenerate HMAC on each page load or session — treat it like a short-lived session token derived from your backend auth.
</Callout>

## Dashboard auth

The web dashboard uses JWT sessions (email/password, magic link, GitHub, or Google OAuth) with optional MFA. Team members get role-based access: Owner, Admin, Developer, or Marketer.

## Key rotation

Roll secret keys from **Settings → API Keys**. Old keys can be deprecated without downtime if you rotate both server and HMAC generation together.
