title: Production Checklist
description: Verify your Nexus Signal integration before switching to Production keys.
---

Use this checklist before routing live traffic to Production.

## Keys & environments

- [ ] Production secret key (`sk_prd_`) stored in secrets manager — not in git
- [ ] Staging tested with `sk_stg_` and real provider keys
- [ ] Public key (`pk_prd_`) in frontend env vars only

## Providers

- [ ] All required providers connected and **test connection** passed
- [ ] Weight splits configured if using multiple providers per layer
- [ ] Provider unit costs set for [cost analytics](/docs/platform/features/cost-analytics)
- [ ] Optional monthly budget configured with alert email verified

## Workflows

- [ ] Templates promoted to **Production Active** (not Draft)
- [ ] Workflows marked **active**
- [ ] Failover paths tested via Development chaos mode
- [ ] Delivery windows set for user-facing marketing sends
- [ ] Topics assigned and preference matrix tested

## SDK integration

- [ ] HMAC generated server-side per session
- [ ] Node SDK circuit breaker / retry handling in place
- [ ] Subscriber `externalId` matches your auth user IDs

## Compliance & ops

- [ ] GDPR purge process documented (subscriber delete API)
- [ ] Team roles assigned (Owner, Developer, Marketer)
- [ ] MFA enabled for admin accounts
- [ ] Audit log retention and monitoring plan defined

## Billing

- [ ] Plan tier matches expected volume
- [ ] Spend cap preference configured
- [ ] Stripe billing portal access for admins

## Go live

1. Switch dashboard environment to **Production**
2. Deploy backend with `sk_prd_` secret
3. Send a canary trigger to internal team
4. Monitor **Audit Logs** and **Overview** for first hour

<Callout type="warn">
Development sandbox does **not** run in Production. Every trigger sends through real providers and incurs provider charges.
</Callout>
