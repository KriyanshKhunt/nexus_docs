---
title: コスト削減
description: プレゼンス抑制、サンドボックステスト、重み付きルーティングでプロバイダー費用を削減。
---

Nexus Signal には、配信性を犠牲にせずに**不要なプロバイダー費用を削減**するためのいくつかの機能が含まれています。

## プレゼンス抑制

サブスクライバーが**アクティブな WebSocket** 接続（今あなたのアプリを使っている）を持っている場合、Nexus は高コストな外部チャンネルをスキップして**アプリ内のみ**で配信できます。

### デフォルト動作

チャンネルノードで、ユーザーがオンラインのときにこれらのチャンネルが抑制されます:

- Web プッシュ
- モバイルプッシュ
- SMS
- WhatsApp

**アプリ内**は常に WebSocket で配信します。これを明示的に制御するにはチャンネルノードで `suppressWhenOnline` を有効にします。

### 仕組み

1. React SDK がハートビート付きの WebSocket を維持
2. Redis が `presence:sub:{subscriberId}`（45秒 TTL）を追跡
3. ワーカーが外部チャンネルをディスパッチする前にプレゼンスをチェック
4. スキップされた配信はライフサイクルステップ `PRESENCE_SUPPRESSION` で `SKIPPED_BY_PREFERENCE` をログ記録

**結果:** ユーザーがすでにあなたのアプリを見ているときに冗長な $0.05 の SMS なし。

## サンドボックスモード（開発）

**開発**環境では、アウトバウンドプロバイダー呼び出しが**仮想シミュレーター**に置き換えられます。API 費用なしにダッシュボードでプッシュ、SMS、メールのプレビューを確認できます。

[サンドボックスモード](/docs/platform/features/sandbox)をご覧ください。

## 重み付きプロバイダールーティング

チャンネルごとに複数のプロバイダーアカウントを実行し、パーセンテージでトラフィックを分割します:

```json
{ "sendgrid": 70, "resend": 30 }
```

**インテグレーション → ウェイトスライダー**で設定します。ワーカーが1〜100をロールしてプロバイダーを選択 — 段階的なマイグレーションと負荷分散に便利です。

## サーキットブレーカーフェイルオーバー

プロバイダーが繰り返し失敗（60秒で5エラー）すると、サーキットが開放されてトラフィックが正常な代替またはワークフローフェイルオーバーブランチにルーティングされます — 死んだエンドポイントへの無駄なリトライ費用を回避。

## 組み合わせ効果

チームは通常、次の方法で費用を削減します:

1. アクティブユーザーへの冗長なプッシュ/SMS を抑制
2. 本番送信前にサンドボックスでテスト
3. 障害のあるプロバイダーから自動的にルーティングを変更
