Follow these steps to trigger a workflow from your backend.

## 1. Create a workspace

Sign in at the [dashboard](https://app.nexussignal.dev). On first login you'll create a workspace and receive **three isolated environments**, each with its own API key pair:

| Environment | Key prefix | Use for |
|-------------|------------|---------|
| Development | `pk_dev_` / `sk_dev_` | Local testing + sandbox simulator |
| Staging | `pk_stg_` / `sk_stg` | QA and integration tests |
| Production | `pk_prd_` / `sk_prd_` | Live traffic |

## 2. Create a workflow

In the dashboard go to **Workflows** and create a workflow named `user.signup` (or any event name). Add at least one **Channel** node (e.g. Email) and publish a template to **Production**.

## 3. Connect a provider

Open **Integrations**, add your email provider (SendGrid or Resend), and paste your API key. Nexus encrypts keys at rest — they are never resold or used for other tenants.

## 4. Install the Node SDK

```bash
npm install @nexus-signal/node
```

## 5. Trigger from your server

```ts
import { NexusClient } from '@nexus-signal/node';

const nexus = new NexusClient({
  secretKey: process.env.NEXUS_SECRET_KEY!, // sk_dev_… or sk_prd_…
  baseUrl: 'https://api.nexussignal.dev',
});

const result = await nexus.workflows.trigger({
  workflowName: 'user.signup',
  recipients: [
    {
      externalId: 'user_123',
      email: 'alex@acme.io',
    },
  ],
  data: { name: 'Alex' },
});

console.log(result.ok ? 'Queued' : result.error);
```

A successful call returns **HTTP 202** with `{ accepted: true, results: [...] }`.

## 6. Verify delivery

Check **Audit Logs** in the dashboard. In Development, messages appear in the **Sandbox** simulator instead of hitting real providers.
